3Tables & columns ~3h

Create the seven custom tables. Build in this order — dependent tables come last so lookups resolve cleanly.

3.1
Creation order
  1. helix_resource — no lookups yet (managedBy added in 3.2)
  2. helix_project — depends on helix_resource (projectManager, teamManager)
  3. helix_phase — depends on helix_project & helix_resource
  4. helix_dependency — depends on helix_phase
  5. helix_allocation — depends on helix_project, helix_resource
  6. helix_scenario — depends on helix_project
  7. helix_scenarioallocation — depends on helix_scenario, helix_resource
  8. helix_timeentry (Phase 2 work — skip for MVP)
3.2
Self-reference on helix_resource

After helix_resource is created, add the helix_managedby lookup pointing back to helix_resource. Maker portal supports this via Add column → Type: Lookup → Related table: Resource.

4Field Security Profile ~30 min

The "Helix Finance" profile enforces Admin-only access to financial columns. Without it, any user querying Dataverse directly (Power BI, Excel) can see budgets and rates regardless of app logic.

4.1
Enable field security on protected columns

For each column below, set Advanced options → Enable column security = Yes. This locks the field at the platform level.

TableColumn
helix_resourcehelix_hourlyrate
helix_projecthelix_budget
helix_projecthelix_funding
helix_projecthelix_spent
helix_projecthelix_fundinggap
helix_scenariohelix_baselinebudget
helix_scenariohelix_estimatedcostdelta
4.2
Create the Field Security Profile

In Power Platform admin center → Environments → Helix-DEV → Settings → Users + permissions → Field security profiles → New:

FieldValue
NameHelix Finance
DescriptionRead access to financial columns across Helix tables

Open the profile → Field permissions tab. For each secured field, set:

  • Read = Allowed
  • Update = Allowed (for budget/funding only; leave Read-only for spent/fundinggap)
  • Create = Allowed (for budget/funding/rate only)
4.3
Add Admin users to the profile

Profile → Users tab → Add users. Add only users who will hold the Helix Admin role. Everyone else will see null for these fields automatically.

✓
Test it now: Sign in as a non-Admin user and query helix_project via the WebAPI. helix_budget should return null. If you see actual values, the profile isn't applied — check field security is enabled on the column itself.

5Security roles ~1h

Four Dataverse security roles map 1:1 to Helix access levels. Row-level scoping uses Business Unit and User ownership.

5.1
Create the four security roles

In Power Platform admin center → Environments → Helix-DEV → Settings → Users + permissions → Security roles → New role. Create each role with the table privileges below.

RoleRead scopeWrite scope
Helix AdminOrganizationOrganization
Helix ManagerBusiness UnitBusiness Unit (allocations only)
Helix PMUser (own + cascaded from owned projects)User-owned projects only
Helix MemberUser (own resource + own allocations)None

Apply these privileges to every helix_* table. Use the Core records tab on the role for the helix tables.

5.2
Assign Helix Admin to Field Security Profile

Users with Helix Admin must also be members of the Helix Finance FLS profile. The security role alone doesn't grant field access — both layers are required.

7Connection references ~30 min

Connection references decouple the Code App from environment-specific connection IDs. Required for clean DEV → TEST → PROD promotion.

7.1
Create the Dataverse connection reference

In your Helix solution → New → More → Connection reference:

FieldValue
Display nameHelix Dataverse
Namehelix_dataverse
ConnectorMicrosoft Dataverse
ConnectionPick your active connection (or create new)
!
Never hardcode connection IDs in power.config.json or anywhere in the Code App. Reference only the schema name (helix_dataverse). Hardcoded IDs are the #1 cause of solution import failures across environments.
7.2
Wire up power.config.json
// power.config.json { "name": "Helix", "displayName": "Helix · IT Portfolio & Resource OS", "connectionReferences": { "helix_dataverse": { "connectorId": "shared_commondataserviceforapps", "dataSourceName": "helix_dataverse" } }, "environmentVariables": {} }
⚠
dataSourceName must be lowercase — capitalized values cause PAC to silently fail the push without an error message.

10User onboarding ~1h ongoing

10.1
Create helix_resource records

For each IT department user, create a row in helix_resource:

  • Set helix_systemuser to their Dataverse user record (1:1 link)
  • Set helix_accesslevel to Admin / Manager / PM / Member
  • Set helix_jobrole to their functional role
  • Set helix_capacityhoursperweek (40 default; 20–32 for part-time)
  • Set helix_hourlyrate (Admin can do this; FLS hides from others)
  • Set helix_managedby to their manager's helix_resource record
10.2
Assign security roles

In Power Platform admin center → Users → [select user] → Manage security roles, assign the matching Helix role (Admin / Manager / PM / Member). For Admins, also add to the Helix Finance Field Security Profile.

Known gotchas Critical reading

Production lessons. Most cost half a day each the first time.

#1PAC CLI v2.5.1 pac code push is broken

The push command silently exits with a 0 status code but doesn't actually upload your build. Use npx power-apps push from the @microsoft/power-apps npm package instead, or downgrade to PAC v1.30.0. Microsoft hasn't published a fix as of this writing — check release notes before assuming a newer version resolves it.

#2window.powerAppsBridge timing

The Power Platform bridge that exposes the current user (window.powerAppsBridge) is not always available on first render. Wrap your useCurrentUser hook with a retry-on-undefined pattern using setTimeout, or use TanStack Query's retry with a backoff. Without this, your role gates flicker between "Member" and the actual role on every refresh.

#3Full vs short connection IDs

When PAC outputs the connection reference info, it shows both a full GUID and a short ID. The Code App needs the short form in power.config.json. Mixing them causes 401 errors at runtime that look like auth problems but are actually wiring problems.

#4Lowercase dataSourceName

PAC normalizes dataSourceName to lowercase when generating the config. If you handwrite the config with PascalCase, PAC will silently overwrite or fail. Always lowercase.

#5Build-time env substitution only

Vite environment variables are baked in at build time, not runtime. If you change VITE_* values between DEV and TEST, you must rebuild — not just redeploy. Use connection references for things that vary by environment; reserve VITE_* for feature flags only.

#6FLS profile membership is not exported

The Helix Finance FLS profile is part of the solution, but the list of users assigned to it is not. After importing the solution to TEST or PROD, you must manually re-add the Admin users to the profile in the target environment. Add this to your deployment checklist.

#7Alternate key on solution export

Maker portal sometimes drops the alternate key on helix_allocation when exporting the solution. Open the exported XML and confirm <EntityKey> is present. If missing, re-add via PAC: pac solution add-component --component-type 14 --component helix_uniqueallocation.
